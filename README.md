# AionUi

AionUi is an inherited Electron and WebUI application currently identified as version `1.7.0`. Its package metadata describes a graphical interface for command-line AI agents, model providers, local files, document previews, and browser-accessible workspaces.

This repository is under **fork and distribution review**. It has not yet been approved as an upstream mirror, maintained fork, or independently distributed derivative, and no aevespers2 binary, update channel, supported-platform claim, signing identity, or privileged integration is authorized.

## A.L.I.S.T.A.I.R.E. role candidate

AionUi is being evaluated as an optional **human-facing host shell** for an accepted QSO-STUDIO review contract. Under the current candidate boundary:

- QSO-STUDIO owns the domain-neutral review model and compatibility fixtures;
- AionUi may render that contract through a separately admitted adapter;
- Repository `1` or an approved successor remains the authority source for dispositions, capabilities, revocations, corrections, and recovery state;
- human approval remains a separate immutable record;
- display, selection, authentication, annotation, export, delivery, or execution success is not approval or canonical acceptance.

No review adapter has been implemented or approved.

## Runtime modes

The inherited repository exposes multiple materially different surfaces:

| Mode | Current status |
|---|---|
| Static GitHub Pages console | Documentation-only, public, read-only, credential-free |
| Electron desktop | Inherited implementation; baseline and security acceptance pending |
| Local WebUI | Inherited implementation; authentication and local-network profile pending |
| Remote WebUI | Blocked pending deployment, TLS, proxy, session, privacy, incident, and recovery architecture |
| Packaging and updates | Blocked pending fork identity, reproducibility, signing, provenance, distribution, and rollback evidence |

A permission or feature present in one mode does not transfer automatically to another.

## Current gates

Before an application candidate is eligible, the repository must have:

1. approved upstream/fork/derivative identity and exact provenance;
2. approved product name, first supported platform, distribution channel, and owners;
3. clean dependency installation, lint, format, tests, and one reproducible platform build;
4. Electron, WebUI, filesystem, process, provider, credential, storage, updater, accessibility, and supply-chain review;
5. accepted QSO-STUDIO/AionUi contract boundary and runtime-mode policies;
6. shared positive and fail-closed compatibility fixtures;
7. SBOM, checksums, signing status, provenance, incident response, recovery, and rollback evidence;
8. explicit human approval for the exact immutable candidate.

## Scope boundary

The documentation candidate changes no inherited Electron/WebUI behavior and grants no credential, provider, agent, filesystem, repository-write, device-control, approval, capability, payment, release, signing, deployment, remote-administration, or canonical-state authority.
